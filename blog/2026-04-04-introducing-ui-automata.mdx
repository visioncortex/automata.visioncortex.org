There is a category of work that is too interactive to automate with a traditional script, and too tedious to keep doing by hand. Checking a service status in Event Viewer. Extracting a value from a legacy enterprise app. Configuring a settings dialog that has no API. These tasks require judgment (but not much of it). They are exactly the kind of thing you would hand off to a capable colleague.

AI agents can be that colleague. The problem is the interface.

## Why Windows Desktop Automation Is Hard

The Windows desktop is genuinely difficult to automate programmatically. Unlike the web (where the DOM is structured, accessible, and designed to be read by code), the Windows desktop is a patchwork of UI frameworks: Win32, MFC, WPF, UWP, WinUI 3, embedded web views, custom renderers. Each one exposes its element tree differently. Elements can appear, disappear, and rebuild mid-interaction. Dialogs pop up unexpectedly. Applications behave differently across OS versions, display scaling settings, and language packs.

Vision-based computer use is a compelling answer to this complexity. Rather than trying to understand every framework, you just look at the screen. It works, and for tasks that require genuine visual reasoning (reading a chart, handling a UI with no accessibility support at all) it remains the right tool.

But for automation at any scale, it carries real costs. Each step is a round-trip to an inference API. Pixel coordinates shift when the window moves or the DPI changes. Every intermediate state the agent needs to observe is a separate inference call. And a sequence of screenshots is not an auditable record: when something goes wrong, there is no structured trace to reason about.

UI Automata takes a complementary approach: use the semantic layer that is already there.

## A Declarative Workflow Engine

</video>

Instead of asking an AI model to guess where to click from pixels, UI Automata gives agents a structured interface to the Windows UI through Windows UI Automation (UIA): the same accessibility layer that screen readers use. UIA queries are local, return semantic data, and take milliseconds.

You describe what should happen in YAML. The engine handles the rest.

```yaml
- intent: click the Save button
  action:
    type: Click
    scope: main_window
    selector: ">> [role=button][name=Save]"
  expect:
    type: DialogPresent
    scope: main_window
  timeout: 10s
```

Every step has three parts: an **action** (what to do), an **expect** (what state the UI should be in afterwards), and optionally a **recovery** (what to do if something unexpected happens). The engine executes the action, then polls the expect condition every 100ms. When it passes, it moves on. When the timeout expires, it checks recovery handlers before failing cleanly.

No sleeps. No guessing. No silent failures.

## The Innovation: CSS-Like Selectors for Windows UI

The Windows UIA element tree is messy. A typical application window contains hundreds of elements: nested panels, unlabelled containers, toolbars with identical-looking buttons, lists where every row has the same role and no name. No single property is reliably enough on its own to identify the exact element you need.

We built a CSS-like selector language that lets you combine every available signal (role, name, AutomationId, structural position, parent context, tree depth) into a precise, unambiguous address.

```python
>> [role=button][name=Save]                    # role AND name
>> [id=TabListView] > [role=tab item]:first    # first child of the tab list
>> [role=button][name=Settings]:parent         # navigate up from a known child element
>> [role=dialog][name='Confirm Save As'] >> [role=button][name=Yes]  # button inside a specific dialog
```

This is not cosmetic. It solves real problems:

- `[name^=Don][name$=Save]` matches "Don't Save" whether the apostrophe is `U+0027` or `U+2019` (right single quotation mark, common in newer Windows). An exact match would fail on one of them.
- `:parent` and `:ancestor(n)` let you anchor on a row container you can only identify through a landmark element inside it (a common pattern in dense professional applications).
- `>` vs `>>` pins you to a specific tree depth. When there are nested toolbars, `> ToolBar` means the one at the top level, not whichever one `>>` finds first.

Selectors operate on UIA semantic properties (role, accessible name, AutomationId), not pixel coordinates. They survive resolution changes, theme changes, localization, and most application updates. A selector that worked last week still works today.

## Under the Hood: The Shadow DOM

Windows UI Automation is a cross-process RPC protocol. Every element query is a round-trip to the target process. Walk a path of nested elements (window → panel → toolbar → button) and each step is a separate cross-process call. A workflow naively issuing fresh UIA queries on every step can spend most of its time waiting for element resolution, not doing work.

UI Automata solves this with what we call the **shadow DOM**: a cached mirror of the live element tree. The engine resolves each anchor once, records the live element handle, and reuses it for every subsequent step. No re-traversal. A cached handle lookup is effectively free compared to a cross-process UIA query.

The shadow DOM also handles identity. When a Root anchor is first resolved, the engine records the exact HWND (the OS-level window handle) of the matched window. Every subsequent resolution goes directly to that HWND, bypassing the selector entirely. If the application changes its title during navigation, the anchor does not drift. If a new window with a similar title opens, it is ignored. A Root anchor is not "the window that currently matches this selector." It is *this specific window, forever*.

When handles go stale (because a panel reloaded or a toolbar refreshed), the engine walks up the element tree to the nearest live ancestor and re-queries downward from there. The anchor is silently re-resolved, and the step proceeds as if nothing happened.

This is the inverse of React's virtual DOM. React maintains a virtual tree to efficiently *write to* a UI it controls. The shadow DOM maintains a virtual tree to efficiently *read from* a UI it does not control.

## Universal Coverage

Windows applications are not a single thing. A modern Windows machine might run Win32 apps from 1998, MFC-based industrial software, WPF enterprise tools, UWP apps hosted by ApplicationFrameHost, WinUI 3 apps with tabbed interfaces, terminal windows, and a web browser, all at once.

UI Automata covers all of them.

| Toolkit | Notes |
|---|---|
| Win32 | Classic API; dialogs are real child windows; stable across OS versions |
| MFC | Dense, deeply nested UIA trees; shadow DOM caching makes this viable |
| WPF | Named AutomationIds on most controls; virtualizing panels require `Invoke` |
| UWP | Hosted by ApplicationFrameHost; anchor by window title, not process name |
| WinUI 3 | Direct process; tabbed interfaces; inline prompts instead of child dialogs |
| Web browser | CDP for page content; UIA for browser chrome and download flows |

Where UIA is unavailable (terminal windows, some custom renderers), the engine can fall back to pure vision. Vision and structured automation are not mutually exclusive; they complement each other. The typical pattern: use vision during authoring to explore the UI and draft selectors, then use UIA in workflow runs.

## Built for AI Agents

UI Automata is not a scripting framework that an agent happens to use. It is designed from the ground up to be the interface between an agent and the Windows desktop.

Every step carries an `intent` field: a human-readable description of what it is trying to do. The engine logs every step with its outcome, producing a structured trace the agent can read and reason about. When something goes wrong, the trace tells the agent exactly what state the UI was in at the moment of failure, which recovery handlers ran, and what they found.

The MCP interface exposes everything an agent needs:

- `desktop`: inspect live element trees and test selectors against any window
- `app`: list installed applications and launch them
- `browser`: CDP page inspection and navigation
- `workflow`: run a workflow file, receive structured output
- `vision`: OCR and layout for windows that lack UIA support
- Schema reference and built-in linter: allow agents to author workflow scripts proficiently

An agent can explore an unfamiliar UI, author workflow steps interactively, run them immediately, observe the result, and promote working steps into a reusable workflow file. The entire loop (explore, script, verify, save) happens in a conversation. No separate GUI, no visual designer, no specialist required.

## Industrial-Grade Applications

One of the hardest problems in desktop automation is professional software: CAD tools, legacy ERP systems. These applications have dense, deeply nested UIA trees with hundreds of elements per panel, virtualizing lists where off-screen rows have no bounding box, and toolbars with dozens of identical-looking buttons distinguishable only by AutomationId.

This is exactly where traditional automation tools break down. Pixel-coordinate approaches fail because nothing is stable. Imperative scripts fail because the application state is too complex to reason about procedurally.

UI Automata's combination of precise selectors, shadow DOM caching, and declarative recovery handling makes these applications tractable. The `Invoke` action calls `IInvokePattern::Invoke()` directly (bypassing the bounding box entirely), so it works on off-screen list rows and toolbar buttons that `Click` cannot reach. The `:parent` and `:ancestor` navigators let you anchor on a row container you can only identify through an element inside it.
